过去几个月，Anthropic 在产品层面的动作非常密集。Claude Code 从内测到正式发布，Claude.ai 的 Projects 和多智能体功能持续迭代，整个产品矩阵的轮廓越来越清晰。本文尝试从产品定位和技术架构两个维度，系统梳理这两类产品的本质区别和演进方向。

---

## 一、产品定位：谁在用，用来做什么

### Claude Code：给工程师的终端 AI

Claude Code 本质上是一个 **CLI 工具**，运行在开发者自己的终端里。它不是一个对话框，而是一个可以真正操作文件系统、执行命令、读写代码的 AI 代理。

区别于 GitHub Copilot（代码补全）或 ChatGPT（对话式建议），Claude Code 的核心卖点是 **执行力**：它不仅能理解你的意图，还能直接动手——修改文件、运行测试、提交 Git、调用外部服务。一次对话可以跨越几十个文件、几百行代码，完成一个完整的功能开发或 bug 修复闭环。

目标用户非常明确：**专业软件工程师**。它假设用户有能力审查变更、理解 diff、判断 AI 行为是否正确。这也是为什么它默认以 CLI 形式交付，而不是一个低代码 GUI。

### Claude 协作（Projects + 多智能体）：给团队和业务的 AI 工作流

Claude.ai 的 Projects 功能面向的是更广泛的场景——知识工作者、跨职能团队、业务流程自动化。核心能力不是"写代码"，而是 **跨对话的上下文持久化**和**任务分解协作**。

Projects 让 Claude 能记住一个项目的背景（文档、规范、历史决策），在不同对话中保持连贯。多智能体模式则允许一个"编排者"（Orchestrator）将复杂任务拆解后分发给多个专门的子智能体并发执行，最后汇总结果。

两者的定位差异可以用一句话概括：

> Claude Code 是**工具**，帮工程师提效；Claude 协作是**工作流**，帮组织提效。

## 二、技术架构拆解

### 2.1 Claude Code 的架构

```
┌─────────────────────────────────────────────────────────┐
│  用户终端（Claude Code CLI）                              │
│                                                          │
│  ┌──────────┐   ┌──────────┐   ┌──────────────────────┐ │
│  │ 对话上下文 │   │  工具层   │   │   Hooks / 权限系统   │ │
│  │ (压缩管理) │   │          │   │                      │ │
│  └──────────┘   │ • Read   │   │ • 自动允许白名单       │ │
│                 │ • Write  │   │ • 用户确认提示         │ │
│                 │ • Edit   │   │ • pre/post 钩子        │ │
│                 │ • Bash   │   └──────────────────────┘ │
│                 │ • Glob   │                             │
│                 │ • Grep   │                             │
│                 └──────────┘                             │
│                      │                                   │
│  ┌───────────────────▼──────────────────────────────┐   │
│  │             子智能体系统（Agent 工具）              │   │
│  │  Orchestrator → [SubAgent1] [SubAgent2] [SubAgentN]│  │
│  └────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
                        │  API 调用
                        ▼
              Anthropic Claude API
              (Sonnet / Opus / Haiku)
```

**上下文管理**

Claude Code 的核心工程挑战之一是 **上下文窗口管理**。长时间的开发任务很容易让上下文膨胀到模型限制附近。它的应对策略：

- **渐进式压缩（Compaction）**：当上下文接近限制时，自动将历史对话压缩成摘要，保留关键决策和文件状态，释放 token 空间。
- **惰性读取**：不预先把整个代码库塞进上下文，而是按需读取——用 Glob 定位文件，用 Grep 定位符号，用 Read 读取目标片段。
- **工具结果截断**：超长的命令输出会被截断，避免单次工具调用污染整个上下文。

**工具层设计**

工具是 Claude Code 与外部世界交互的唯一接口，设计上遵循几个原则：

1. **可审计性**：每次工具调用对用户可见，不藏行为。
2. **最小权限**：默认只允许读操作，写操作和 Bash 执行需要用户授权或配置到白名单。
3. **幂等优先**：Edit 工具用精确字符串匹配而不是行号，避免并发修改时的位置漂移。

**子智能体（Subagent）系统**

当任务可以并行分解时，Claude Code 可以用 `Agent` 工具启动独立的子智能体。每个子智能体：

- 拥有独立的上下文（不共享父智能体的对话历史）
- 可以访问相同的工具集
- 结果通过单条消息返回给编排者

这个设计的好处是 **上下文隔离**——子任务的噪声不会污染主任务，同时天然支持并行执行。

**MCP（Model Context Protocol）**

MCP 是 Anthropic 推出的开放协议，让 Claude Code 能接入第三方工具服务器。任何人都可以写一个 MCP Server，暴露工具给 Claude 调用——无论是数据库查询、API 请求、还是专有系统集成。

```
Claude Code
    │
    ├── Built-in Tools (Read/Write/Bash/...)
    │
    └── MCP Servers（通过 stdio 或 HTTP/SSE）
            ├── 数据库 MCP Server
            ├── GitHub MCP Server
            ├── Jira MCP Server
            └── 自定义内部系统
```

MCP 的意义在于：它把工具生态从 Anthropic 一家公司的内部问题，变成了一个开放的社区问题。

### 2.2 Claude 协作（多智能体）的架构

```
┌─────────────────────────────────────────────────────────┐
│                    编排层（Orchestrator）                  │
│                                                          │
│  任务输入 → 分解 → 调度 → 合并 → 输出                     │
│                    │                                     │
│         ┌──────────┼──────────┐                         │
│         ▼          ▼          ▼                         │
│    [SubAgent A] [SubAgent B] [SubAgent C]               │
│    市场调研     财务分析     竞品对比                      │
│         │          │          │                         │
│         └──────────┼──────────┘                         │
│                    ▼                                     │
│             结果汇总 → 综合报告                           │
└─────────────────────────────────────────────────────────┘
```

**Projects 的持久化机制**

Projects 的核心是 **项目级上下文**：用户上传的文档、历史对话摘要、项目规范，构成一个持久的知识库。每次对话开始时，这个知识库会被注入上下文，让 Claude 不需要重新了解背景。

技术上，这类似于 RAG（检索增强生成）的简化版——差别在于 Projects 的上下文是结构化管理的，而不是纯粹基于向量检索。

**多智能体的协调模式**

目前 Claude 多智能体主要有两种协调模式：

| 模式 | 特点 | 适用场景 |
|------|------|----------|
| **串行链**（Chain） | A 完成后结果传给 B | 有依赖关系的顺序任务 |
| **并行扇出**（Fan-out） | 编排者同时派发多个子任务 | 独立可并行的子任务 |

实际系统中两者混合使用。编排者本身也是一个 Claude 模型实例，它的任务是：理解全局目标、判断任务依赖关系、合理分配子智能体、整合结果并处理异常。

## 三、关键技术：提示缓存（Prompt Caching）

多智能体和长上下文场景都面临一个经济性问题：每次 API 调用都要传输大量重复的上下文（系统提示、项目文档、工具定义），token 成本极高。

Anthropic 的 **Prompt Caching** 机制直接解决这个问题：

```
第一次请求：[System Prompt 5000 tokens] + [User Message]
             → 写入缓存（5min TTL）
             → 按正常价格计费

第二次请求（5分钟内）：[System Prompt 5000 tokens] + [User Message]
             → 命中缓存
             → 缓存 token 成本降低约 90%
```

在 Claude Code 的 `/loop` 模式（持续执行循环任务）或多智能体并发场景中，缓存命中率直接决定了系统是否有商业可行性。这也是为什么 Claude Code 的调度逻辑会特别注意保持在 300 秒（缓存 TTL）以内唤醒。

## 四、未来趋势

### 4.1 从"辅助"到"自主"

当前 AI 编程工具的主流模式还是 **人在回路**（Human-in-the-loop）：AI 提建议，人来决策和确认。Claude Code 已经支持无人值守的后台执行，但工程师仍然是最终审批者。

下一步演进方向是 **渐进式自主**：AI 获得更大的决策权，人的角色从"逐步审批"变成"设定目标 + 异常处理"。技术上需要解决的问题：

- **更精确的置信度估计**：AI 知道自己在哪里确定、哪里不确定，只在不确定时请求人工介入。
- **更细粒度的权限系统**：不是简单的"允许/拒绝"，而是"在特定范围内允许自主，超出范围后请求确认"。
- **可验证的执行轨迹**：每一步操作留下可审查的记录，出问题时可以溯源和回滚。

### 4.2 MCP 生态的标准化

MCP 现在还处于早期，各种工具的实现质量参差不齐。随着生态成熟，预计会出现：

- **MCP 工具市场**：类似 npm / PyPI，发布和发现高质量的 MCP Server。
- **企业级 MCP 网关**：统一管理权限、审计日志、速率限制，让企业可以安全地把内部系统暴露给 AI。
- **跨模型互操作**：MCP 不绑定 Claude，其他模型也可以接入相同的工具生态。

### 4.3 智能体记忆的标准化

目前每个智能体框架都有自己的记忆实现（向量库、文件、数据库），缺乏标准。未来可能出现类似 MCP 的协议，标准化智能体的记忆读写接口——让记忆成为可插拔的基础设施，而不是每个 agent 框架各自为政。

### 4.4 多模态工具使用

Claude 已经支持图像输入，但工具使用层面的多模态还不完善。未来的方向是：

- **屏幕操作**（Computer Use）：Claude Code 可以直接操作 GUI 应用，而不只是命令行工具。
- **视觉调试**：上传截图，AI 直接分析 UI 问题并修改对应代码。
- **图表理解**：读懂架构图、流程图，把视觉信息转化为可操作的代码变更。

## 五、我的判断

Claude Code 和 Claude 协作不是竞争关系，而是 **两个不同入口**：一个从代码工具切入，一个从工作流切入，最终指向同一件事——让 AI 成为团队里真正能交付工作的成员，而不只是一个高级搜索引擎。

对工程师来说，现在值得投入时间的方向：

1. **掌握 Claude Code 的工具系统**：理解它如何管理上下文、何时用子智能体、如何配置权限，能让你更有效地驾驭它，而不是被它驾驭。
2. **关注 MCP 生态**：把自己日常用的工具封装成 MCP Server，是一个一次投入、长期受益的基础建设。
3. **设计"AI 友好"的代码库**：清晰的目录结构、一致的命名、CLAUDE.md 里写好项目规范，AI 的输出质量与代码库的可读性高度相关。

AI 代理的能力还在快速演进，但它放大的始终是工程师的判断力和架构能力。工具越强，设计思考就越重要。
